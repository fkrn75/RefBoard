## 핵심 근본원인 (1건이 P1+P2를 동시 유발)

`main.ts`가 노트/드로잉 노드에 `scene.getSprite()`를 오용한 것이 가장 큰 단일 원인이었다.

- `scene.getSprite()`(scene.ts)는 **`Sprite` 인스턴스일 때만** 반환하고, 노트(`Text`)·드로잉(`Graphics`)
  id를 넘기면 `undefined`를 반환한다.
- 따라서 z순서 갱신·노트 편집 시 숨김 처리가 **이미지에만 적용되고 노트/드로잉은 누락**됐다.
- 타입 무관인 `scene.getNode()`로 교체하면 P1(z-order 미반영)과 P2(노트 이중 표시)가 동시에 해결된다.

> 비교용 접근자: `getNode()`=타입 무관, `getSprite()`=Sprite 전용, `getNote()`=Text 전용, `getDrawing()`=Graphics 전용.

### 보류 1건 상세 — 공유 만료(expired) 화면

`supabase-share.ts`의 `load()`는 만료 보드를 `forbidden`/`auth-required`로 분류한다. 이는 의도된 한계다.

- `boards_read` RLS 정책이 `(expires_at is null or expires_at > now())`를 select 조건에 포함하므로
  (`docs/share-backend-design.md` §5 참조), **만료된 보드는 행 자체가 숨겨져** 미존재·미허가·만료가
  모두 동일하게 0행으로 합쳐진다.
- anon key로는 이 RLS를 우회하는 별도 존재확인 쿼리를 만들 수 없고(모든 select가 같은 정책을 탐),
  만료여도 행을 받는 건 소유자뿐인데 소유자에겐 만료 화면이 불필요하다.
- 정확히 식별하려면 RLS를 타지 않는 `SECURITY DEFINER` RPC
  (예: `board_block_reason(id) → 'expired'|'forbidden'|'not_found'`)가 백엔드에 필요하다.
- 0행을 무조건 expired로 표시하는 부정확한 추측은 보안 무해하나 오인 UX를 낳으므로
  (정확성 > 커버리지) 채택하지 않는다. `viewer/main.ts`의 `expired` 분기는
  `LocalShareAdapter`/임베드 경로 및 향후 RPC 도입 대비로 유지한다.

  (이미지 외 노드는 직렬화·공유에서 일관되게 필터).

8. 성능 모드 4단계
9. 최근 파일 열기 UI (`recent.ts`는 기록만, UI 미연결)
10. 리치 텍스트 노트 (현재는 평문 + 색/크기/글꼴까지만)
